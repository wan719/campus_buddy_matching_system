### 1.2 技术实现要点

#### 功能1：RBAC数据库表结构设计

**实现思路**：
- 遵循RBAC权限模型标准，设计5张核心表：users、roles、permissions、user_role、role_permission
- 同时整合校园搭子系统的业务表：activity、team_member、application、review
- 使用外键约束保证数据完整性，设置ON DELETE CASCADE级联删除
- 为常用查询字段创建索引，提升查询性能

**关键代码片段**（核心表结构）：
```sql
CREATE TABLE `users` (
    `id` BIGINT NOT NULL AUTO_INCREMENT,
    `student_id` VARCHAR(20) NOT NULL UNIQUE,
    `username` VARCHAR(50) NOT NULL UNIQUE,
    `password` VARCHAR(255) NOT NULL,
    `email` VARCHAR(100) NOT NULL UNIQUE,
    `enabled` BOOLEAN DEFAULT TRUE,
    PRIMARY KEY (`id`)
);

CREATE TABLE `user_role` (
    `user_id` BIGINT NOT NULL,
    `role_id` BIGINT NOT NULL,
    PRIMARY KEY (`user_id`, `role_id`),
    FOREIGN KEY (`user_id`) REFERENCES `users`(`id`) ON DELETE CASCADE,
    FOREIGN KEY (`role_id`) REFERENCES `roles`(`id`) ON DELETE CASCADE
);
```

**遇到的难点及解决**：
- 难点：如何将RBAC模型与现有业务表（activity等）无缝整合
- 解决方案：统一使用`users`表作为用户主表，业务表通过外键关联；保留原有字段（student_id、credit_score等）确保业务兼容

#### 功能2：实体类设计（整合业务方法）

**实现思路**：
- 使用Lombok注解（`@Data`、`@Builder`）简化代码
- User实体类增加`roles`字段实现多对多关系
- 实现业务方法：`addRole`、`hasRole`、`getPermissions`、`hasPermission`
- 通过角色间接获取用户权限，实现权限继承

**关键代码片段**（权限获取方法）：
```java
public Set<String> getPermissions() {
    if (this.roles == null) {
        return Set.of();
    }
    return this.roles.stream()
            .flatMap(role -> role.getPermissions().stream())
            .map(Permission::getName)
            .collect(Collectors.toSet());
}
```

**遇到的难点及解决**：
- 难点：避免JSON序列化时的循环引用（User↔Role双向关联）
- 解决方案：在关联字段上使用`@JsonIgnore`注解，只序列化单向关系

### 3.2 典型问题深度分析

**问题描述**：
运行`./gradlew test`时，出现`ClassNotFoundException: cn.edu.swu.campus_buddy_matching_system.model.UserTests`，测试无法执行。

**问题分析**：
- Gradle在编译测试代码时，无法找到测试类的字节码文件
- 可能原因：测试类文件不存在，或放置的目录结构与包名不匹配
- 检查发现`UserTests.java`文件确实缺失，之前只提供了代码但未在正确路径创建文件

**解决过程**：
1. 尝试1：在项目根目录搜索`UserTests.java`文件（结果：不存在）
2. 尝试2：检查`src/test/java`目录结构，发现`model`子目录不存在（结果：需要创建）
3. 最终方案：创建完整的目录结构`src/test/java/cn/edu/swu/campus_buddy_matching_system/model/`，将`UserTests.java`放入该目录，并确认包名与路径一致

**经验总结**：
- Java项目的测试类必须放在`src/test/java`目录下，且包名必须与目录结构完全一致
- 编写测试代码后，应立即运行`./gradlew test`验证，而不是等到最后才执行
- 使用`--stacktrace`参数可以获取更详细的错误信息，便于定位问题
