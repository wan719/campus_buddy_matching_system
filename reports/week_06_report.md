### 1.2 技术实现要点

#### 功能1：RBAC 管理接口实现

**实现思路**：
- 按照 RESTful API 风格，基于 Spring Boot 实现认证模块、用户管理模块、角色管理模块和权限管理模块。
- 采用 Controller → Service → Mapper 的分层架构，将请求接收、业务逻辑和数据库访问职责分离。
- 使用 MyBatis 注解与 XML 结合实现简单查询和复杂关联查询，例如角色详情与权限列表、用户详情与角色列表。

**关键代码片段**：
```java
@GetMapping("/{id}")
public ApiResponse<RoleResponse> getRoleById(@PathVariable Long id) {
    return ApiResponse.success("查询成功", roleService.getRoleById(id));
}
```

**遇到的难点及解决**：
- 难点：MyBatis 中 resultMap 在注解和 XML 中重复定义，导致启动时报错。
- 解决方案：保留注解版 resultMap，删除 XML 中重复定义的同名 resultMap，统一通过完整限定名复用映射。

---

#### 功能2：测试与接口文档完善

**实现思路**：
- 使用 JUnit5 + Mockito 为 Controller 层和 Service 层编写单元测试。
- 测试覆盖成功场景与失败场景，例如“查询成功”“资源不存在”“删除成功”等。
- 为各 Controller 补充 Swagger 注解，接入 springdoc 生成 Swagger UI 页面，便于展示和接口调试。

**关键代码片段**：
```java
@Test
void testGetUserById_NotFound() {
    when(userService.getUserById(999L))
            .thenThrow(new IllegalArgumentException("用户不存在"));

    IllegalArgumentException ex = assertThrows(
            IllegalArgumentException.class,
            () -> userController.getUserById(999L)
    );

    assertEquals("用户不存在", ex.getMessage());
}
```

**遇到的难点及解决**：
- 难点：最初项目中没有引入 Swagger 依赖，Controller 中 Swagger 注解无法导入。
- 解决方案：在 `build.gradle` 中补充 springdoc 依赖，并统一给 4 个 Controller 添加接口说明注解。

### 3.2 典型问题深度分析

**问题描述**：  
在实现权限模块和角色模块时，项目启动失败，日志提示 `Result Maps collection already contains value for ...permissionResult`，应用在创建 `sqlSessionFactory` 阶段直接报错。

**问题分析**：  
通过排查 Mapper 接口和 XML 文件，发现 `PermissionMapper.java` 中已经使用 `@Results(id = "permissionResult")` 定义了一个 resultMap，而 `PermissionMapper.xml` 中又再次定义了同名 `resultMap`。MyBatis 在启动时会同时注册这两个映射，导致命名冲突并抛出异常。

**解决过程**：
1. 尝试1：直接修改 XML 中字段映射（结果：失败）  
   仅修改字段名无法解决 resultMap 重复注册的问题。
2. 尝试2：保留 XML resultMap，删除注解 resultMap（结果：未采用）  
   会影响现有注解查询方法，改动范围较大。
3. 最终方案：保留 Java 注解版 resultMap，删除 XML 中重复定义的同名 resultMap，并在 XML 中通过完整限定名引用（结果：成功）

**经验总结**：  
通过这次问题我进一步认识到，在 MyBatis 中混用注解和 XML 时，必须提前统一好映射定义的边界，避免同名 resultMap 重复注册。后续开发时，应先确定简单查询使用注解、复杂查询使用 XML，并保证 namespace 与方法引用保持一致。
