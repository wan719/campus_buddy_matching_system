# 校园搭子系统2.0：理论课标准与公司实战流程对照及融合方案

## 一、理论课标准流程（系统分析与设计课程）

这是大学教材中经典的**结构化方法**，通常分为以下阶段：

`1. 系统规划 → 2. 系统分析 → 3. 系统设计 → 4. 系统实施 → 5. 系统运行与维护`

### 各阶段核心产出物：

| 阶段 | 核心活动 | 产出物 |
| :--- | :--- | :--- |
| **1. 系统规划** | 可行性研究、项目计划 | 可行性分析报告、项目计划书、进度表 |
| **2. 系统分析** | 需求获取、需求建模 | 需求规格说明书、业务流程图、数据流图（DFD）、用例图 |
| **3. 系统设计** | 总体设计、详细设计 | 架构设计文档、模块设计、数据库设计（ER图）、接口设计、类图、时序图 |
| **4. 系统实施** | 编码、测试 | 源代码、测试用例、测试报告 |
| **5. 运行维护** | 部署、维护 | 部署文档、用户手册、维护记录 |

## 二、公司实战开发流程（以互联网公司为例）

公司里很少按理论课的五个阶段线性推进，而是采用**敏捷开发** + **DevOps** 的迭代模式：

`需求池 → Sprint规划 → 开发 & 测试 → Code Review → 合并 & 部署 → 复盘`
`          ↑_______________迭代_______________↓`

### 各环节核心活动：

| 环节 | 核心活动 | 常用工具/方法 |
| :--- | :--- | :--- |
| **需求管理** | 用户访谈、需求评审 | Jira / 飞书 / Notion，用户故事、PRD |
| **技术方案** | 方案评审、技术选型 | 技术设计文档（TDD）、架构图 |
| **开发阶段** | 编码、单元测试 | Git Feature Branch、代码规范 |
| **测试阶段** | 功能测试、集成测试 | 测试环境、Postman自动化、JUnit |
| **Code Review** | 代码审查 | GitHub PR、SonarQube |
| **部署上线** | CI/CD、灰度发布 | Jenkins / GitHub Actions、Docker |
| **运维监控** | 日志监控、告警 | 阿里云监控、ELK、Sentry |

## 三、对比分析：理论 vs 实战

| 维度 | 理论课标准 | 公司实战 | 差异说明 |
| :--- | :--- | :--- | :--- |
| **流程形态** | 瀑布式，线性推进 | 敏捷迭代，螺旋式 | 实战中需求会不断变更，不会等全部分析完才设计 |
| **文档要求** | 文档详尽，作为评分依据 | 文档精简，够用就好 | 公司强调“可工作的软件”胜过“详尽的文档” |
| **角色分工** | 一人扮演多角 | 产品、开发、测试、运维分离 | 实战中协作更多，沟通成本高 |
| **时间投入** | 分析占30%-40% | 分析占10%-20%，持续迭代 | 公司追求快速验证市场 |
| **建模工具** | 必画DFD、ER图、类图 | 灵活使用，常用ER图、流程图、时序图 | DFD在公司几乎不用，用业务流程图替代 |
| **验收方式** | 老师评分 | 产品经理验收、用户反馈 | 实战中“上线”才是真正的验收 |

## 四、融合版：校园搭子系统2.0开发全流程（推荐）

结合两者的优点，我为你定制一套**既符合课程要求，又贴近实战**的流程：

### 📋 阶段一：系统规划（1-2天）
**目标**：回答“做不做”和“怎么做”

| 任务 | 实战做法 | 课程对应产出 |
| :--- | :--- | :--- |
| 项目背景分析 | 写一段话说明为什么要做这个系统 | 可行性分析报告 |
| 目标用户定义 | 画出用户画像 | 项目计划书 |
| 技术选型 | Spring Boot + MySQL + MyBatis | 技术选型说明 |
| 制定里程碑 | 规划3个Sprint，每个Sprint 1周 | 项目进度表 |

### 📊 阶段二：系统分析（2-3天）
**目标**：搞清楚“要做什么”

| 任务 | 实战做法 | 课程对应产出 |
| :--- | :--- | :--- |
| 需求调研 | 采访5-10个同学，收集痛点 | 需求调研记录 |
| 用户故事 | 写出15-20个用户故事（之前我们列的那些） | 用户故事列表 |
| 优先级排序 | MoSCoW法（Must/Should/Could/Won't） | 优先级矩阵 |
| **业务流程图** | 用Draw.io画出核心业务流程 | 业务流程图 |
| **用例图** | 画出用例图（学生、管理员） | 用例图 |
| **数据流图** | 课程要求：画0层、1层DFD | 数据流图 |

### 🏗️ 阶段三：系统设计（2-3天）
**目标**：确定“怎么做”

| 任务 | 实战做法 | 课程对应产出 |
| :--- | :--- | :--- |
| **架构设计** | 画出系统架构图（分层架构） | 架构设计文档 |
| **数据库设计** | 画出ER图 → 逻辑模型 → 物理模型（SQL） | ER图、表结构设计 |
| **接口设计** | 用Apifox定义RESTful API | API文档 |
| **类图** | 画出核心类图（User、Activity等） | 类图 |
| **时序图** | 画出登录、申请加入的时序图 | 时序图 |
| **详细设计** | 写出关键逻辑的伪代码 | 详细设计文档 |

### 💻 阶段四：系统实施（迭代进行）
**目标**：把设计变成可运行的代码

**Sprint 1：用户模块**
| 任务 | 技术实践 |
| :--- | :--- |
| 项目初始化 | Spring Boot + MyBatis + JWT配置 |
| 用户注册/登录 | BCrypt加密、Token生成 |
| 单元测试 | JUnit + Mockito |
| Code Review | GitHub PR |

**Sprint 2：活动模块 + 匹配模块**
| 任务 | 技术实践 |
| :--- | :--- |
| 活动CRUD | MyBatis动态SQL |
| 筛选搜索 | 分页 + 条件查询 |
| 申请加入 | 并发控制（乐观锁） |
| 集成测试 | Postman自动化测试集 |

**Sprint 3：评价模块 + 部署**
| 任务 | 技术实践 |
| :--- | :--- |
| 评价功能 | 信用分自动计算 |
| 部署上线 | 云服务器 + Docker |
| 验收测试 | 产品经理/用户验收 |

### 📝 阶段五：文档交付（贯穿全程）
**目标**：产出课程要求的最终文档

| 文档类型 | 内容 | 完成时间 |
| :--- | :--- | :--- |
| 需求规格说明书 | 用户故事、功能列表、非功能需求 | 阶段二结束 |
| 系统设计说明书 | 架构图、ER图、类图、时序图、接口文档 | 阶段三结束 |
| 测试报告 | 测试用例、测试结果 | 阶段四结束 |
| 用户手册 | 操作指南、常见问题 | 阶段五结束 |
| 项目总结 | 经验教训、后续规划 | 项目结束后 |
